# GodlingBattle 文档唯一性约定

## 目的

避免 `docs/` 根目录与 `docs/superpowers/` 出现同一信息多处维护，降低后续状态更新混乱。

## 唯一来源规则

1. **项目全局进度、最新改动、下一步建议**
- 唯一来源：`docs/HANDOFF.md`
- 其他文档只允许引用，不重复写完整进度列表。

2. **产品目标、边界、规则（相对稳定的信息）**
- 唯一来源：`docs/项目概览.md`
- 不在此文件维护阶段进度与提交记录。

3. **阅读路径与计划入口**
- 唯一来源：`docs/实施计划导读.md`
- 只做导航，不重复维护执行状态。

4. **设计规格（Why/What）**
- 唯一来源：`docs/superpowers/specs/*.md`
- 设计更新只改 specs，不同步复制到 root 文档。

5. **实施任务与步骤（How）**
- 唯一来源：`docs/superpowers/plans/*.md`
- 计划文档维护任务定义与阶段细节，不承担“全局状态公告”职责。

## 更新流程（必须遵守）

- 更新执行进度时：只改 `docs/HANDOFF.md`。
- 变更产品目标/边界时：只改 `docs/项目概览.md`。
- 新增阶段计划时：只在 `docs/superpowers/plans/` 新增或更新对应计划文档。
- 调整阅读入口时：只改 `docs/实施计划导读.md`。

## 冲突处理

若多个文档描述不一致，以以下优先级为准：

1. `docs/HANDOFF.md`（全局进度）
2. `docs/superpowers/specs/*.md`（设计定义）
3. `docs/superpowers/plans/*.md`（实施细节）
4. `docs/项目概览.md` 与 `docs/实施计划导读.md`（摘要与导航）
