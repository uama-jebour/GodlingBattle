# GodlingBattle Observe Phase15 可读性增强设计

## 1. 背景与问题

当前 Observe 界面在真实回放场景存在三类可读性问题：

1. 战场单位在高密度时仍可能出现视觉重叠，影响战局判断速度。
2. 单位死亡后仍长期停留在战场，造成“已死亡但仍占场”的认知负担。
3. 右侧信息区（存活名册 + 战斗日志）层次不够清晰，整体字号偏小，阅读成本高。

本设计基于 2026-03-31 讨论结论，作为 Phase15 的执行依据。

## 2. 目标与边界

### 2.1 目标

1. 让战场 token 在 1920x1080 下保持稳定、无重叠、可快速扫读。
2. 将死亡单位改为“短暂残影后消失”，兼顾反馈感与画面清爽。
3. 保持右侧“两区结构”（上：存活名册；下：战斗日志），但强化分区和可读性。

### 2.2 非目标

1. 不改主流程：`出战前准备 -> 自动观战 -> 结果结算 -> 返回出战前准备`。
2. 不改 runtime 契约字段（`battle_setup` / `battle_result`）。
3. 不引入手动战斗交互。
4. 本轮不要求全分辨率自适配，仅以 1920x1080 为验收主目标。

## 3. 方案选择

采用“槽位化战场方案”：

1. 同侧单位使用固定槽位布局，彻底规避重叠。
2. 死亡单位保留短暂残影（linger 窗口）后从战场移除。
3. 右侧继续“上名册/下日志”语义，日志区拆分关键事件与普通日志层次。

不采用“最小补丁方案”与“混合映射方案”，原因：

1. 最小补丁易产生后续边角重叠回归。
2. 混合映射调参与维护成本高于本阶段收益。

## 4. 架构与组件边界

### 4.1 战场布局层

文件：`scripts/observe/battlefield_layout_solver.gd`

职责：

1. 输入每帧单位快照，按 side 分组后映射到稳定槽位。
2. 输出显示坐标（写回 snapshot 的 `position` 字段），保证同侧 token 不重叠且在 bounds 内。
3. 当 `BattlefieldPanel` 尺寸异常时回退到安全 bounds，避免渲染失败。

约束：

1. runtime 原始 `position` 仅用于排序参考，不直接决定最终显示坐标。
2. solver 只负责布局，不承担视觉样式或生命周期逻辑。

### 4.2 单位生命周期层

文件：`scripts/observe/observe_screen.gd`

新增/强化三态：

1. `alive`：正常显示。
2. `death_linger`：死亡后保持原槽位并显示“已阵亡”标记，直到 `linger_until_tick`。
3. `removed`：超过 linger 窗口后移除 token（`queue_free` + `_token_views` 清理）。

关键行为：

1. seek/jump 到任意帧时重建死亡状态，保证回放一致性。
2. 若出现“复活”数据（`alive: false -> true`），清理历史 linger 并恢复正常态。

### 4.3 右侧信息展示层

文件：`scenes/observe/observe_screen.tscn`、`scripts/observe/observe_screen.gd`

结构保持不变：

1. `AliveRosterPanel`（上半区）展示当前存活单位。
2. `BattleLogPanel`（下半区）展示分层日志。

展示策略：

1. 日志区先显示“关键事件段”（倒下、未响应后果、未响应结算）。
2. 再显示“普通日志段”（最近 N 条战术明细）。
3. 空状态必须有明确文案，避免留白造成“是否出错”的误判。

兼容策略：

1. 继续隐藏 legacy `EventPanel/EventPanelBg`，防止双轨信息冲突。
2. 保持 `get_*_text` 兼容方法可调用，避免测试与外部引用回归。

## 5. 数据流设计

每帧处理顺序：

1. 从 timeline 读取 `_current_entities`。
2. 构建 token snapshot（含 `alive/side/hp/position/visual_flags`）。
3. 经 solver 进行槽位化布局，得到稳定位置。
4. 同步 token 视图并应用生命周期清理规则。
5. 更新右侧名册（仅存活）与日志（关键优先 + 普通补充）。

死亡 linger 规则：

1. 触发时机：单位首次从存活变为死亡。
2. 显示位置：保持死亡时所在槽位。
3. 结束条件：`current_tick > linger_until_tick` 后移除战场 token。

## 6. 可读性规范（1920x1080）

本轮由“测试最低可过”升级为“产品可读”：

1. 四象限标题字号不低于 `24`。
2. 名册正文与日志正文字号不低于 `20`。
3. HUD 中帧号字号不低于 `44`，事件与战技文本字号不低于 `30`。
3. 名册行间距与日志段落间距增加，优先减少信息挤压感。

说明：实现计划需要把以上阈值固化为常量，并由对应测试守卫。

## 7. 错误处理与回退

1. 布局回退：bounds 无效时使用默认安全区域。
2. 数据回退：缺失 `position` 时按 side + 稳定顺序分配槽位。
3. UI 回退：右侧任一子节点缺失时不崩溃，返回空文本并记录 warning。
4. 回放一致性：seek/jump 后重算 linger 索引，避免残影状态错乱。

## 8. 测试与验收

### 8.1 战场布局

1. 扩展 `tests/observe_battlefield_non_overlap_test.gd`，覆盖高密度与边界场景。
2. 验收：同侧 token 矩形不相交，且全部在 bounds 内。

### 8.2 死亡生命周期

新增/扩展测试，覆盖：

1. 死亡当帧进入 linger 并显示标记。
2. linger 窗口内仍可见。
3. 超时后从 `_token_views` 与场景树移除。
4. seek/jump 后状态与目标帧一致。

### 8.3 右侧信息与可读性

1. 名册仅展示存活单位。
2. 日志面板存在“关键事件段 + 普通日志段”两段结构。
3. 1920x1080 下标题与正文达到本轮可读字号阈值。

### 8.4 回归基线

1. 必跑：`tests/observe_*.gd`
2. 建议：`tests/*.gd` 全量回归

## 9. 风险与缓解

1. 风险：槽位化降低“原始位置感”。
   - 缓解：保留 side 分区与排序稳定性，确保战局关系可辨识。
2. 风险：linger 与 seek/jump 组合出现状态污染。
   - 缓解：以目标帧重建生命周期索引，不依赖历史播放状态。
3. 风险：字号提升后发生文本裁切。
   - 缓解：同步调整容器最小高度、滚动区域与自动换行策略。

## 10. 里程碑定义（Phase15）

Phase15 完成标准：

1. 视觉上消除同侧 token 重叠。
2. 死亡单位表现为“短暂残影后消失”。
3. 右侧“名册+日志”在 1080p 下实现清晰分区与易读文本。
4. Observe 测试集通过，且无主流程回归。

## 11. 执行协作约定（Sub-Agent）

当 Phase15 的子任务交由 sub-agent 执行时，采用单轮验收模式：

1. `实现 -> 基础验证 -> 集成`。
2. 不要求执行“两轮代码审核”与“两轮质量审核”。
3. 仅在高风险改动（破坏性操作、数据迁移、安全敏感逻辑、benchmark 门禁策略变更）时再升级审核强度。
