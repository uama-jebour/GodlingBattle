# GodlingBattle 观战战报中心与结果页中文化设计

## 1. 背景与问题

当前版本虽然已经完成基础中文化与观战信息扩展，但仍存在三类关键体验问题：

1. 结果页仍可能出现英文 ID（如 `enemy_wandering_demon`）。
2. 观战日志能看到事件，但用户难以快速理解“这一帧发生了什么”和“战局整体如何变化”。
3. 日志信息层级混杂，缺少“速览”和“深入”两层阅读模式。

目标是把观战右侧面板升级为统一的“战报中心”，并彻底消除结果页英文项目残留。

## 2. 目标与非目标

### 2.1 目标

1. 结果页所有展示文本统一中文，不显示 `hero_ / ally_ / enemy_ / strat_ / battle_` 这类 ID。
2. 观战页提供双层日志：
   - 默认 `简报层`（高可读、低负担）
   - 可展开 `战术明细层`（谁在何时触发了什么）
3. 保持现有观战交互能力（暂停、跳帧、时间线缩放/密度、事件标记跳转）。
4. 不破坏现有 runtime 结算与数据协议，优先在 UI 层做可读性转换。

### 2.2 非目标

1. 本阶段不做“全量数值调试日志”（伤害精确值/每次冷却细节的开发级输出）。
2. 不重写 battle runtime 核心结算流程。
3. 不引入多页面新流程，仍在现有 Observe/Result 页面内完成升级。

## 3. 方案选择

采用“战报中心化”（方案2）：

1. Observe 右侧统一为“战况总览 + 本帧简报 + 可展开战术明细 + 关键时间线 + 标记列表”。
2. 新增 UI 层“日志转换层”，将 runtime 日志转换为用户可读中文句式。
3. 结果页与观战页共用“显示名解析器”，统一处理 ID 到中文名的映射与兜底。

选择理由：

- 改动集中在 UI 消费层，风险显著低于 runtime 协议重构。
- 能一次性解决“英文残留 + 日志不直观”两个核心问题。
- 后续扩展（新增事件类型/日志类型）成本低。

## 4. 信息架构

## 4.1 Observe 右侧战报中心

顺序与职责：

1. `战况总览`
   - 胜负状态、进度帧、双方存活数、阵亡数、触发事件数、战技施放总数。
2. `本帧动态（简报）`
   - 仅显示当前帧关键行为（战技施放、事件预警/结算、倒下）。
3. `战术明细（可展开）`
   - 展示本帧或筛选范围内更完整叙述：
     - 谁施放了什么战技
     - 哪个事件进入预警/结算
     - 结算是否已响应
     - 哪个单位倒下
4. `事件时间线`
   - 保留缩放与密度控制，作为全局节奏导航。
5. `事件标记列表`
   - 保留点击跳帧。

## 4.2 Result 页面结构

1. 胜负标题
2. 战果摘要（存活/阵亡/事件/策略/关键施放）
3. 配置快照（英雄/友军/战技/关卡/种子）
4. 操作按钮（再战、返回）

要求：所有摘要字段均输出中文显示名。

## 5. 数据转换设计

## 5.1 显示名解析器（共享）

新增独立解析器（建议：`scripts/ui/display_name_resolver.gd`），提供：

1. `unit_name_from_unit_id(unit_id)`
2. `unit_name_from_entity_id(entity_id)`（自动剥离末尾数字序号）
3. `strategy_name(strategy_id)`
4. `event_name(event_id)`
5. `battle_name(battle_id)`

兜底策略：

1. 能查 content 就显示中文名。
2. 查不到时显示中文兜底文案（如“未知敌方单位”），不回退英文 ID 原文。

## 5.2 日志转换层（Observe 专用）

新增转换器（建议：`scripts/observe/battle_report_formatter.gd`）：

输入：`log_entries` + `current_tick` + `filter_type`  
输出：

1. `brief_lines`：简报句子数组
2. `detail_lines`：战术明细句子数组
3. `stats`：事件数/战技数/倒下数等统计

事件类型文案映射：

1. `event_warning` -> 事件预警
2. `event_resolve` -> 事件结算（已响应/未响应）
3. `event_unresolved_effect` -> 未响应后果
4. `strategy_cast` -> 战技施放
5. `ally_down/hero_down/enemy_down` -> 友军倒下/英雄倒下/敌方倒下

## 6. 文案规范

### 6.1 简报层模板

1. `第{tick}帧 战技施放：{strategy_name}`
2. `第{tick}帧 事件结算：{event_name}（已响应）`
3. `第{tick}帧 敌方倒下：{unit_name}`

### 6.2 战术明细模板

1. `第{tick}帧｜我方施放【{strategy_name}】`
2. `第{tick}帧｜事件【{event_name}】进入结算｜结果：已响应`
3. `第{tick}帧｜目标【{unit_name}】倒下`

### 6.3 结果页字段规范

统一中文标签：

1. 存活单位
2. 阵亡单位
3. 触发事件
4. 触发战技
5. 关键施放
6. 配置快照

禁止出现英文 key/id 前缀：`hero_ / ally_ / enemy_ / strat_ / battle_`。

## 7. 交互规则

1. 默认显示简报层，明细层默认折叠。
2. 展开明细后：
   - 与当前帧联动刷新
   - 与事件筛选联动
3. 时间线与标记跳转后：
   - 当前帧、简报、明细、总览同步更新
4. 筛选只影响“日志展示”，总览默认显示全局战况统计。

## 8. 错误处理与兜底

1. 缺失字段时不报错，使用中文兜底句：
   - `未知事件`、`未知战技`、`未知单位`
2. 无关键日志时显示：
   - `本帧动态：暂无关键事件`
3. 无结果数据时总览显示：
   - `战况总览：数据准备中`

## 9. 测试策略

新增与强化测试：

1. `tests/display_name_resolver_test.gd`
   - 覆盖 unit/entity/strategy/event/battle 解析与兜底。
2. `tests/result_localization_no_english_ids_test.gd`
   - 覆盖结果页禁止英文 ID 输出。
3. `tests/observe_battle_report_brief_detail_test.gd`
   - 覆盖简报/明细双层日志与展开行为。
4. 更新现有 Observe/Result 测试
   - 确保时间线筛选、跳帧、摘要联动无回归。

门禁：

1. 全量 `tests/*.gd` 必须通过。
2. 新增文本断言：关键 UI 文本不包含 `hero_ / ally_ / enemy_ / strat_ / battle_`。

## 10. 实施顺序

1. 抽取显示名解析器并在 Result 接入（先清英文）。
2. 抽取 Observe 日志转换层（先保证简报可读）。
3. 加入明细展开区与联动。
4. 补齐回归测试并跑全量验证。

该顺序保证每步都能独立验证并保持可回退。
