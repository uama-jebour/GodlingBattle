# GodlingBattle 观战四象限战斗界面设计（Phase14 预研）

## 1. 背景

当前 `Observe` 页面以“播放条 + 右侧事件面板”为主，和目标“战斗主画面优先”的产品体验差异较大。  
本设计将 `Observe` 升级为四象限战斗界面，并保持现有 runtime 与回放能力不变。

## 2. 目标与范围

### 2.1 目标

1. 左上为主战场大区域，突出实时战斗状态。
2. 左下为技能策略横向卡片，包含触发反馈与冷却可视化。
3. 右上为敌我双方存活单位双列清单（名称 + 当前生命值）。
4. 右下为可追溯战斗日志（预告、施法、效果、死亡）。
5. UI 字体整体可读，不出现“字太小看不清”。
6. 敌方单位显示具体名称，不再出现“敌方单位”泛称。

### 2.2 非目标

1. 不改 battle runtime 的结算规则与数据协议。
2. 不新增手操战斗交互（仍为自动观战）。
3. 不改 Result 场景业务流程。

## 3. 已确认方案（用户确认）

采用“方案1：`ObserveScreen` 原地改造”。

### 3.1 对比结论

1. 原地改造（选中）：复用现有播放/时间线/日志链路，风险最低。
2. 新建 `BattleHud` 再接入：结构更优但接线成本更高。
3. 全新场景替换：自由度最高但回归风险最大。

## 4. 布局设计

## 4.1 总体分区

1. 横向：`左 68% : 右 32%`
2. 左侧纵向：`上 64% : 下 36%`
3. 右侧纵向：`上 25% : 下 75%`（用户最终确认）

## 4.2 四象限职责

1. 左上（主战场）  
   渲染单位、血条、名称、受击/施法动效，承载核心观看焦点。
2. 左下（技能策略区）  
   横向技能卡，展示触发态与冷却填充。
3. 右上（存活清单）  
   双列展示我方/敌方存活单位名与当前生命值。
4. 右下（战斗日志）  
   时间序事件流，支持战局全过程追溯。

## 5. 视觉与交互规格

## 5.1 主战场单位可读性

1. 单位布局采用“阵营分区 + 槽位化（slot） + 最小间距”策略，禁止重叠。
2. 单位名称始终显示在单位上方，血条紧随单位展示。
3. 名称来源统一走 `display_name_resolver`，敌方显示具体名（如“追猎魔”）。
4. 单位死亡瞬间必须出现明确“死亡标识”（如 skull/icon + `已阵亡` 标签），并在短时停留后再按现有规则移除或淡出。

## 5.2 主战场动效（用户确认强度）

1. 受击动效：短时抖动 + 亮度变化。
2. 被施法动效：状态环脉冲（buff/debuff 标记）。
3. 飘字反馈：伤害值、状态词（如“减速-35%”）。
4. 技能束线：施法方向或命中关系的短时光束提示。

说明：动效强度维持当前原型 V3，不再额外增强。

## 5.3 技能策略区

1. 卡片横向排列，展示技能名与冷却文本（当前值/总冷却）。
2. 触发时卡片高亮与轻脉冲提示（“已触发”态）。
3. 冷却用“自下而上填充色块”表示进度，并实时更新。

## 5.4 存活清单

1. 双列结构：我方 / 敌方。
2. 每行显示：单位名 + 当前生命值（数值）。
3. 右上“存活清单”只显示存活单位；死亡单位从该清单移除（死亡信息由右下日志承担）。
4. 死亡标识不在“存活清单”内承载，而在主战场死亡单位与日志条目中显式展示。

## 5.5 战斗日志

1. 日志包含四类核心事件：  
   事件预告、技能施放、效果结算、单位死亡。
2. 典型文案要求：
   - “追猎魔将在 5 秒后出现”
   - “圣堂天使施放【寒潮冲击】→ 追猎魔群体减速 35%”
   - “虚环猎犬C死亡（击杀者：裂隙恶魔）”
3. 日志按时间顺序持续追加，支持滚动浏览与回溯。

## 6. 字体与可读性规范

1. 主标题/分区标题：不小于 `15px`。
2. 正文与关键信息：不小于 `13px`，优先 `14px`。
3. 关键数字（HP、冷却、伤害）需具备对比度和字重。
4. 目标分辨率 `1920x1080` 下可稳定阅读，不依赖放大。

## 7. 数据流与模块边界

## 7.1 数据输入（沿用现有）

1. 时间线帧：`timeline[*].entities`
2. 结果日志：`last_battle_result.log_entries`
3. 显示名解析：`display_name_resolver`

## 7.2 UI 侧新增/调整职责

1. `ObserveScreen`：负责布局装配与回放时钟驱动。
2. `TokenView`：负责单位绘制、血条与局部动效触发。
3. 技能卡渲染层：消费策略运行时状态，输出触发态与冷却进度。
4. 存活清单渲染层：由当前帧实体快照计算敌我存活与HP。
5. 日志渲染层：继续消费 `battle_report_formatter` 输出可读文案。

## 7.3 兼容性要求

1. 现有播放能力必须保留：暂停、跳帧、倍速、时间轴定位。
2. 时间轴功能可弱化视觉层级，但不能失效。

## 8. 异常与兜底

1. 名称缺失：显示中文兜底（如“未知敌方单位”）。
2. HP 缺失：显示 `0` 或 `--`，但 UI 不崩溃。
3. 日志缺失：显示“暂无战斗记录”。
4. 动效数据缺失：退化为静态显示，不影响主流程。

## 9. 测试设计

## 9.1 回归重点

1. 布局节点存在与比例断言（四象限结构）。
2. 单位重叠防护（最小间距）断言。
3. 敌方名称不出现“敌方单位”泛称断言。
4. 技能触发态与冷却填充可见性断言。
5. 存活清单正确联动当前帧 HP。
6. 日志完整覆盖“预告/施法/效果/死亡”。
7. 字体最小尺寸断言（关键节点）。
8. 死亡单位出现“死亡标识”并满足最短可见时长断言。

## 9.2 兼容回归

1. 保持 `observe_playback_*`、`observe_event_*` 相关测试通过。
2. 保持 `observe_strategy_cast_hud_test.gd` 与战报中心测试通过。
3. 不引入 `hero_/ally_/enemy_/strat_/battle_` 英文 ID 泄露回归。

4. 切片D：右上存活清单 + 右下日志增强
5. 切片E：测试补齐与回归验证

## 11. 验收标准

1. 四象限布局与比例与本设计一致（含右侧 25/75）。
2. 主战场单位不可重叠，名称/血条可辨。
3. 受击、被施法、死亡标识动效存在且强度与 V3 原型一致。
4. 技能卡具备触发反馈与“下向上冷却填充”。
5. 存活清单和战斗日志满足信息完整性要求。
6. 字体可读性达标，测试全量通过。
