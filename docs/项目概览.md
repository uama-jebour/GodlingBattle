# GodlingBattle 项目概览

## 1. 这是什么项目

`GodlingBattle` 是一个全新的、独立的 Godot 项目，用来实现新的战斗系统。

它与旧的 `Godling` 项目并列存在，目的就是避免：

- 新旧系统脚本混在一起
- 旧项目的 autoload / 数据结构继续污染新项目
- 后续迭代时难以判断哪些逻辑属于新系统

因此，`GodlingBattle` 的基本原则是：

- 项目独立
- 数据独立
- 场景独立
- UI 独立
- 只参考旧项目里已经验证过的自动战斗思路，不整体搬运旧模块

## 2. 当前产品目标

首期目标不是做完整大游戏，而是做一个“可玩的正式风格战斗 Demo”。

玩家进入项目后，直接进入 `出战前准备` 界面，在这里配置：

- 1 名英雄
- 3 名普通单位
- 携带策略
- 测试关卡

配置完成后进入自动观战，战斗结束后进入结果结算页，再返回 `出战前准备`。

主流程固定为：

`出战前准备 -> 自动观战 -> 结果结算 -> 返回出战前准备`

## 3. 当前已确认的体验方向

### 出战前准备

风格定位：

- 庄重神秘的战术陈列台

布局方向：

- 英雄与 3 个普通单位并排陈列
- 策略区是一整条较长的陈列带
- 不做地图预览
- 整体要像正式游戏界面，不像调试工具

### 自动观战

表现方向：

- 2D 棋子
- 血条
- 重视信息可读性
- 重视事件预警与策略触发反馈

### 结果结算页

风格定位：

- 冷静的战术报告页

首屏重点信息：

- 胜负
- 存活单位
- 关键事件触发摘要
- 关键策略触发摘要

## 4. 当前已确认的战斗规则

- 胜利条件：敌方全灭
- 失败条件：英雄死亡，或超过 `600s`
- 仿真频率：`10 tick/s`
- 随机性要求：固定 seed，可复现
- 地图首期只影响寻路
- 首期只做近战 AI / 远程 AI
- 首期不做暴击、护甲、抗性、仇恨等二级属性

补充说明：

- 如果普通友军全部死亡，但英雄还活着，战斗继续
- 首期所有可选内容默认解锁，直接可选
- 首期不做编队预设保存

## 5. 当前已确认的系统边界

首期核心玩法是：

- 战前构筑
- 战中自动验证 build
- 战后复盘

因此首期不做旧项目那种“混合交互战斗”。

原因很简单：

- 它会把系统重心从“构筑验证”拉回“战中手操”
- 会让新项目一开始就分裂成两套 runtime
- 会把旧项目里复杂耦合的战斗 UI 包袱一起带进来

首期规则是：

- 战斗全自动
- 主动策略由 AI / runtime 自动释放
- 玩家不在战斗中手动下达指令
