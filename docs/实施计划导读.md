# GodlingBattle 实施计划导读

## 1. 这份文件的用途

这不是完整技术计划，而是给“明天继续接着干”的快速导读。

如果你没时间重新看整份英文计划，先看这份，就能知道：

- 现在准备做什么
- 第一阶段先做哪些事
- 为什么按这个顺序做

完整计划仍然在：

- [2026-03-30-godlingbattle.md](/Users/uama/Documents/Mycode/GodlingBattle/docs/superpowers/plans/2026-03-30-godlingbattle.md)

## 2. 第一阶段的核心目标

第一阶段不是把所有系统一次做完，而是先把最小可运行主流程做通：

`出战前准备 -> 自动观战 -> 结果结算`

只要这条主流程跑通，就说明新项目方向成立，后面再补内容和打磨体验。

## 3. 推荐执行顺序

### 第一步：搭新项目骨架

先真正把 `GodlingBattle` 变成 Godot 项目，而不是只有文档目录。

需要落地的最小内容：

- `project.godot`
- `icon.svg`
- 主场景
- 基础 autoload
- 最小目录结构

这一步完成后，项目至少能正常启动，不再只是空目录。

### 第二步：建立数据层

先不要急着做复杂编辑器或 Excel 导入。

首期建议直接在项目里放静态内容注册表，先把这些数据建起来：

- 单位
- 策略
- 事件
- 测试关卡
- 测试包

核心原则：

- 新项目自己维护数据
- 不依赖旧项目 `ContentDB`
- 所有 ID 在新项目里重新明确

### 第三步：做 `出战前准备`

这是新项目真正的入口界面。

首期要让它能完成这 4 件事：

- 选英雄
- 选 3 个普通单位
- 选策略
- 选测试关卡

完成后输出一份标准化 `battle_setup`，交给运行时使用。

注意：

- 这个界面是正式游戏风格，不是调试表单
- 但第一版也不用过度精修
- 先把结构、层次、信息组织做对

### 第四步：做最小 battle runtime

这一步是核心中的核心。

第一版 battle runtime 只要先做到：

- 初始化 battle state
- 近战 / 远程自动行为
- 基础攻击结算
- 胜负判定
- 时间上限判负
- 事件预警 / 响应 / 生效的最小闭环

重点不是一开始把所有策略都做复杂，而是先让“跑起来”这件事稳定成立。

### 第五步：接自动观战层

观战层只负责“显示”，不要让它反过来控制战斗规则。

第一版观战层重点：

- 棋子位置
- 血条
- 事件预警提示
- 策略触发提示

重点是看得懂，不是炫技。

### 第六步：做结果结算页

结果页第一版只需要稳稳显示：

- 胜负
- 存活单位
- 关键事件摘要
- 关键策略摘要

然后支持一个动作：

- 返回 `出战前准备`

### 第七步：补测试包和批跑工具

当主流程跑通后，再补：

- 6 套测试包
- CSV 导出
- deterministic 测试

这一步是为了验证系统是否稳定、是否值得继续扩。

## 4. 为什么是这个顺序

因为这个顺序能最早回答最重要的问题：

`这个新项目的主流程成立吗？`

如果先去做：

- 精致视觉
- 大量单位扩充
- 复杂策略系统
- 更大的事件系统

很可能会在主流程都还没跑通时就把时间消耗掉。

所以当前策略是：

- 先打通链路
- 再补内容
- 最后做打磨

## 5. 明天开始时最建议直接做的事情

如果你明天一打开项目就准备继续推进，最建议直接从这 3 件事开始：

1. 在 `GodlingBattle` 下真正创建 Godot 项目基础文件
2. 建最小内容注册表
3. 做 `出战前准备` 场景壳子

只要这三步完成，项目就从“文档阶段”真正进入“可执行阶段”了。
